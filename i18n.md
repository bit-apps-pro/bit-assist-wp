# Bit Assist Translation Build System

This document describes how translations work in the Bit Assist plugin. The plugin uses WordPress's standard i18n system for both backend (PHP) and frontend (React) translations.

## Overview

The translation system extracts translatable strings from PHP files using WP-CLI, merges any extra strings, and generates translation files in multiple formats:

- **`.pot`** (Portable Object Template): Source template for translators
- **`.po`** (Portable Object): Language-specific translations (maintained by translators)
- **`.mo`** (Machine Object): Binary format used by PHP's gettext
- **`.json`** (JED format): JSON format used by WordPress JavaScript i18n

## How It Works at Runtime

### Admin panel (React app)

1. [Layout.php](backend/app/Views/Layout.php) enqueues `wp-i18n` and the main script with handle `bit-assist-index-MODULE`.
2. `wp_set_script_translations('bit-assist-index-MODULE', 'bit-assist', languages/)` tells WordPress to inject translations for that handle.
3. WordPress inlines a `<script>` that calls `wp.i18n.setLocaleData(...)` before the app script loads.
4. The Vite build aliases `@wordpress/i18n` → [frontend/src/wp-i18n-shim.ts](frontend/src/wp-i18n-shim.ts), so every `__()` / `_x()` / `_n()` / `_nx()` call in React code delegates directly to `window.wp.i18n`.
5. WordPress looks for a JSON file named `{domain}-{locale}-{handle}.json` in the `languages/` directory, e.g. `bit-assist-bn_BD-bit-assist-index-MODULE.json`.

### Widget iframe (standalone HTML page)

The widget runs in an iframe at `/iframe/`, which cannot use `wp_enqueue_script`. Instead:

1. [IframeController.php](backend/app/HTTP/Controllers/IframeController.php) manually outputs WordPress's `hooks.min.js` and `i18n.min.js` CDN scripts in the `<head>`.
2. It registers a temporary script handle `bit-assist-widget-script-JAVASCRIPT-ASYNC`, calls `wp_set_script_translations()` on it, then captures the inline translation `<script>` via `wp_scripts()->print_translations($handle, false)` and injects it into the iframe `<head>`.
3. The widget bundle (also built with the same `@wordpress/i18n` alias shim) reads translations from `window.wp.i18n` at runtime.
4. WordPress looks for a JSON file named `bit-assist-{locale}-bit-assist-widget-script-JAVASCRIPT-ASYNC.json`.

### Backend (PHP)

1. `Plugin.php` calls `load_plugin_textdomain('bit-assist', false, 'languages')`.
2. WordPress loads `languages/bit-assist-{locale}.mo`.
3. `__('Hello', 'bit-assist')` returns the translated string.

```mermaid
    A[PHP code with __() calls] -->|wp i18n make-pot| B[bit-assist.pot]

```

## Build Scripts

All commands run from the `frontend/` directory.

### Main Command

```bash
npm run i18n
```

Runs the full pipeline: creates `languages/` if needed, then runs `i18n:pot`.

### Individual Steps

#### 1. Generate POT (master template)
```bash
npm run i18n:pot
```
- Runs from the plugin root (`cd ..`)
- Also scans the `bit-assist-pro` sibling directory (if present) and merges its strings
- Excludes `frontend`, `widget-iframe`, `node_modules`, `vendor`, `scripts-for-translate`
- Outputs to `languages/bit-assist.pot`

#### 2. Compile MO files (for PHP)
```bash
npm run i18n:mo
```
- Compiles `languages/bit-assist-bn_BD.po` → `languages/bit-assist-bn_BD.mo`
- Add more locales by editing this script

#### 3. Generate JSON for JavaScript
```bash
npm run i18n:json
```
- Runs `wp i18n make-json languages/ languages/ --no-purge`
- Produces one JSON per handle referenced in the PO file
- Used by `wp.i18n.setLocaleData()` in both the admin panel and the widget iframe

## Configuration Files

### [`frontend/vite.config.mts`](frontend/vite.config.mts)

Key i18n-related settings:

```js
resolve: {
  alias: {
    '@wordpress/i18n': path.resolve(__dirname, 'src/wp-i18n-shim.ts')
  }
},
build: {
  terserOptions: {
    mangle: { reserved: ['__', '_x', '_n', '_nx', 'sprintf'] }
  }
}
```

- The alias replaces the bundled `@wordpress/i18n` package with a thin shim that delegates to `window.wp.i18n` at runtime.
- `mangle.reserved` prevents Terser from renaming the i18n function names (WP-CLI's string scanner looks for these literal names).

### [`frontend/src/wp-i18n-shim.ts`](frontend/src/wp-i18n-shim.ts)

```ts
export const __ = window.wp.i18n.__
export const _n = window.wp.i18n._n
export const _nx = window.wp.i18n._nx
export const _x = window.wp.i18n._x
export const sprintf = window.wp.i18n.sprintf
```

Every import of `@wordpress/i18n` in React code resolves to this shim, so translations flow through WordPress's native `wp.i18n` object which has already been seeded with the locale data before the script runs.

## Adding Translations

### 1. Add translatable strings to your code

**React/TypeScript:**
```tsx
import { __, _x, sprintf } from '@wordpress/i18n'

// Simple translation
const greeting = __('Hello World', 'bit-assist')

// Translation with context (for disambiguation)
const noun = _x('Post', 'blog post', 'bit-assist')

// Translation with variables
const message = sprintf(__('Welcome %s', 'bit-assist'), userName)
```

> Always pass the `'bit-assist'` text domain as the last argument so WP-CLI can pick it up.

**PHP:**
```php
echo __('Hello World', 'bit-assist');
echo _x('Post', 'blog post', 'bit-assist');
echo sprintf(__('Welcome %s', 'bit-assist'), $userName);
```

### 2. Regenerate the POT template

```bash
cd frontend
npm run i18n
```

This creates/updates `languages/bit-assist.pot`.

### 3. Send POT to translators

Translators use the `.pot` file to create `.po` files for each language:
- `languages/bit-assist-bn_BD.po` (Bengali)
- `languages/bit-assist-es_ES.po` (Spanish)
- etc.

### 4. Compile translations

After receiving `.po` files from translators, run:

```bash
npm run i18n:mo    # Compile to .mo for PHP
npm run i18n:json  # Convert to .json for JavaScript (wp i18n make-json)
```

## Translation Files

### Version Control

**Commit these files:**
- ✅ `languages/bit-assist.pot` (master template)
- ✅ `languages/*.po` (translator-maintained)
- ✅ `languages/*.json` (needed at runtime for JS)

**DON'T commit:**
- ❌ `languages/*.mo` (binary, generated from .po)

### JSON File Naming

WordPress resolves JSON translation files by the convention:

```
{domain}-{locale}-{script-handle}.json
```

| Context | Handle | Example filename |
|---|---|---|
| Admin panel | `bit-assist-index-MODULE` | `bit-assist-bn_BD-bit-assist-index-MODULE.json` |
| Widget iframe | `bit-assist-widget-script-JAVASCRIPT-ASYNC` | `bit-assist-bn_BD-bit-assist-widget-script-JAVASCRIPT-ASYNC.json` |

`wp i18n make-json` generates these automatically from the `.po` files.

### File Formats

#### POT/PO Format
```po
msgid "Hello World"
msgstr "হ্যালো বিশ্ব"

# With context
msgctxt "blog post"
msgid "Post"
msgstr "পোস্ট"

# Plural forms
msgid "One item"
msgid_plural "%d items"
msgstr[0] "একটি আইটেম"
msgstr[1] "%d টি আইটেম"
```

#### JED JSON Format
```json
{
  "domain": "bit-assist",
  "locale_data": {
    "bit-assist": {
      "": {
        "domain": "bit-assist",
        "lang": "bn_BD",
        "plural_forms": "nplurals=2; plural=(n != 1);"
      },
      "Hello World": ["হ্যালো বিশ্ব"]
    }
  },
  "translation-revision-date": "2025-01-01 00:00+0000"
}
```

## Troubleshooting

### Strings not translating

1. **Check locale is set:**
   ```bash
   wp option get WPLANG --allow-root
   ```

2. **Verify JSON files exist with the right handle name:**
   ```bash
   # Admin panel
   ls -la languages/bit-assist-*-bit-assist-index-MODULE.json
   # Widget iframe
   ls -la languages/bit-assist-*-bit-assist-widget-script-JAVASCRIPT-ASYNC.json
   ```

3. **Check browser console** for i18n errors.

4. **Verify locale data is available** in browser console:
   ```javascript
   wp.i18n.getLocaleData('bit-assist')
   ```

5. **Function names mangled?** Ensure `mangle.reserved` in `vite.config.mts` includes all i18n function names. WP-CLI cannot extract strings from minified/renamed function calls.

### Build errors

1. **"No PO files found"** — Normal if no translations exist yet; safe to ignore.
2. **"Invalid PO structure"** — Check `.po` file syntax.
3. **"Directory not found"** — Run `npm run i18n` (it creates `languages/` automatically).
